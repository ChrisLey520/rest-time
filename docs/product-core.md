# 栖时产品核心

本文档回答一个问题：用户到底为什么用栖时，以及后续功能如何保证不偏离这个原因。

## 核心结论

栖时不是通用任务管理工具，也不是报表或账单工具。栖时的核心价值是：

> 当用户想做事但不知道下一段该做什么、怎么开始、如何坚持时，帮用户在 30 秒内进入一段明确、有边界、可完成、可复盘的专注。

所有功能都必须服务这个闭环：选择下一段 -> 开始专注 -> 完成或中断 -> 自动沉淀记录 -> 给出下一段建议。

## 目标用户

优先服务一类主用户：需要自主管理时间的知识工作者和创作者。

他们包括自由职业者、学生、研发、设计、产品、写作者、备考者。共同点不是职业，而是他们每天都要面对以下问题：

- 知道有很多事要做，但打开工具时不知道现在该先做哪一个。
- 容易把任务写得太大，导致迟迟不开始。
- 真正开始后也很容易撑不住，随手退出或切走。
- 可能直接打开娱乐 App，把栖时放后台，让后台计时伪装成完成。
- 做完以后很快忘记时间花在哪里，月底无法复盘。
- 希望变得稳定，但不想维护复杂项目管理系统。

客户账单、月报、导出是扩展价值，只适合在用户已经持续记录后出现。它们不能取代“下一段专注”这个主线。

## 核心 Job

用户雇用栖时完成的任务是：

> 帮我把混乱的一天切成下一段能开始的专注，并让我知道这段时间最后变成了什么。

这个 Job 有三个层次：

1. 开始：我现在该做哪一段？
2. 坚持：这段时间不要被轻易打断。
3. 回收：做完以后自动变成进度、记录、复盘和必要的项目归属。

如果一个功能不能提升这三件事中的至少一件，就不应该成为当前优先级。

## 必须满足的体验

### 30 秒进入

冷启动后用户必须在 30 秒内能做出选择并开始第一段专注。

验收标准：

- 首页必须直接给出下一段建议，而不是让用户先整理完整任务系统。
- 没有任务时必须允许自由专注，同时提供一个低成本的新建任务入口。
- 开始按钮必须始终清晰，不能被报表、Pro、设置、统计内容抢走主视觉。

### 一段必须有边界

用户不是来管理无限待办的，而是来完成下一段。

验收标准：

- 任务必须能被估算为 1 到 N 次专注。
- 当前专注进行时，相关任务不能被随意删除或改坏记录归属。
- 主动结束必须保留为中断记录，但不能计入完成专注。

### 退出必须有救援

默认假设用户没有足够毅力。产品不能指望用户靠自律硬撑完整 25 分钟，而要在退出那一秒提供更小的台阶。

验收标准：

- 开始前必须根据最近完成、放弃和离场记录判断当前风险；高风险用户默认进入 5 分钟保底启动，中风险用户进入 10/15 分钟稳住启动，而不是直接要求完整 25 分钟。
- 启动降档必须向用户解释原因，并保留“仍然开始完整专注”的选择，避免产品变成黑箱。
- 保底启动的风险等级、原因和启动秒数必须进入专注记录和导出，用于复盘“短启动是否把用户救回来了”。
- 专注一开始就必须给出一个 3 分钟内可执行的“下一步小动作”，让用户不用先靠意志力理解完整任务。
- 专注中必须允许用户完成小动作、换成更小动作、或声明“卡住了”，让产品把任务继续拆小。
- 专注中点击结束时，不能一键丢掉本段，必须先出现救援选择。
- 救援选择必须先询问退出原因，并进入短暂冷静期，避免用户连续点击就丢掉本段。
- 冷静期必须提供“回去继续”和“确认终止”两个清晰出口；确认终止即失败，不能把已过时间结算为完成。
- 如果退出原因是“卡住了”，必须优先把当前动作改成更小的 3 分钟动作，而不是只要求用户继续忍耐。
- 已经坚持到最低门槛也不能提前收下，只有计时到点且前台完整的硬专注才可结算，避免奖励系统被冲动退出稀释。
- 用户仍然可以明确放弃，但必须经过冷静期和确认，并记录退出原因用于复盘，而不是随手误触。
- 真正放弃前必须留下一个“退出交接”，把当前小动作转换成下一次可恢复的入口，避免用户空手退出后下次不知道从哪里回来。
- 用户已经放弃后，下一次打开必须出现 1/3/5 分钟恢复阶梯，优先回到原任务，把失败恢复成下一次行动，而不是只展示失败记录。
- 重启完成后必须能追溯来源中断记录，用于复盘“失败后是否回来了”。
- 恢复阶梯可以记录为完成的恢复动作，但不能计入正式任务进度、今日目标、客户账单或完整专注轮次，避免把低谷恢复伪装成交付。

### 后台不能刷专注

默认专注必须是在场专注。用户切到后台或打开其他 App 时，栖时不能继续把时间算成已完成，否则产品会奖励逃避。

验收标准：

- 专注一旦开始不能暂停，只能终止；终止即失败，不能获得奖励。
- 专注中进入后台、切换到非白名单 App、锁屏导致心跳断档或 App 被后台终止时，当前专注必须立即记录为失败，不得恢复为完成。
- App 被杀后重开，不得按墙上时间补记完成；如果生命周期能捕获到销毁，应记录为后台终止风险。
- 专注进行中必须维护在场心跳；心跳长时间中断后恢复，本轮必须失败，不能转成可继续的暂停。
- 回到前台后，用户只能查看失败原因、留下交接，并通过 1/3/5 分钟恢复阶梯重启。
- 服务卡片和提醒必须同步失败/整理状态，避免继续暗示“正在专注”或“已暂停可继续”。
- 专注记录必须保留离开次数、离开时长、守门模式、白名单、完整性状态和违规原因，作为前台专注凭证；这些时间可用于复盘和导出审计，但不能把离开时间计入实际专注时长。
- 系统到点提醒只能提示用户回到 App 收束确认，不能在后台直接宣告“已完成”。

### 时光币奖励

时光币只奖励完整、在场、没有守门违规的正式专注。它的作用是把“真的守住一段时间”变成可见资产，后续可接兑换、权益或成就。

验收标准：

- 任何 5 分钟以上完整守住的正式硬专注都发放时光币——保底启动救回来的一段也不能空手，避免惩罚做对事情的用户。
- 5-30 分钟按分钟计币；30 分钟以上进入加速档，时长越长越多，但必须有单次上限和单日上限，避免刷奖励。
- 用户终止、离开 App、杀后台、心跳断档或恢复阶梯完成，都不能获得时光币。
- 今日时光币进度必须在专注首页可见，用户能知道奖励规则和今日上限。

### 完成必须有回收

完成一段专注后，用户应该感到“这段时间没有丢”。

验收标准：

- 完成后自动更新任务进度、今日目标、连续记录、项目归属。
- 记录页能回答今天做了多久、本周是否稳定、本月投入在哪里。
- 复盘文案必须给下一步建议，而不是只展示数字。
- 完成页必须给出明确选择：歇息、继续下一段、记录复盘或结束今天。

### 复杂能力后置

月报、账单、导出、订阅都是后置能力。它们的出现顺序必须建立在稳定记录之上。

验收标准：

- 记录少于 1 段时，不应强调月报或账单价值。
- 记录少于 3 天时，不应给出过度确定的趋势判断。
- 账单只服务有项目归属的完成记录，不能干扰普通专注用户。

## 功能优先级

### P0：下一段专注闭环

- 首页下一段建议。
- 一键开始自由专注。
- 从任务开始专注。
- 中途退出救援：询问原因、20 秒冷静、继续或确认终止。
- 后台防作弊：离开即失败，不能恢复结算。
- 完成后自动记录和更新任务进度。
- 歇息和继续。
- 今日目标和最小复盘。

P0 的判断标准：用户今天能不能完成第一段，并知道下一段是什么。

### P1：计划和稳定性

- 任务预计专注次数。
- 今日/明日/未计划队列。
- 优先级。
- 一键计划模板。
- 周容量、明日结转、高效时段建议。
- 服务卡片和系统提醒。

P1 的判断标准：用户能不能连续几天把混乱任务转成可执行专注。

### P2：复盘和价值证明

- 周复盘。
- 月度报告。
- 项目投入分布。
- CSV/图片导出。
- 客户账单。

P2 的判断标准：用户能不能看见自己的时间投入已经产生了价值。

### 暂不优先

- 团队协作。
- 复杂项目管理层级。
- AI 自动生成大计划。
- 过早账号体系和云同步。
- 复杂标签系统。

这些功能可能商业化很强，但当前会稀释“下一段专注”的主线。

## 产品护栏

每个新功能进入开发前，必须回答以下问题：

1. 它是否让用户更快开始下一段专注？
2. 它是否让当前专注更不容易中断？
3. 它是否让完成后的记录更有用？
4. 它是否增加了用户开始前的负担？
5. 它是否会让首页偏离“开始下一段”？

如果第 4 或第 5 个问题答案是“是”，功能必须后置、隐藏到记录/设置页，或拆小。

## 当前偏移风险

当前版本已经具备很多高级能力，但存在三个风险：

- 记录页信息密度偏高，客户账单、月报、复盘、历史记录都在抢解释权。
- 设置页已经出现 Pro 和账单参数，新用户可能误以为这是商业管理工具。
- 任务页能力接近轻量项目管理，需要继续保持“估算专注段”而不是扩展成完整项目管理。

修正方向：

- 首页继续只围绕“下一段”和“开始”。
- 记录页按用户阶段分层：无记录看引导，少量记录看今日/本周，长期记录再看月报/账单。
- Pro 能力命名要强调“时间回收”和“复盘交付”，而不是泛泛堆高级功能。

## 当前已落地的核心路径

- 首页无任务时支持“写下一件事并开始”，降低首次进入成本。
- 首页有任务时展示今日计划、下一段建议、待办投入和开始按钮。
- 首页根据最近失败/离场情况展示保底启动卡片，高风险默认从 5 分钟开始，中风险默认从 10/15 分钟开始，并允许手动切回完整专注。
- 专注中展示 3 分钟下一步小动作，支持完成小动作、换更小动作和卡住救援。
- 专注中点击终止会先出现退出原因和 20 秒冷静期，确认终止即失败且无奖励。
- 确认放弃会自动生成退出交接，把当前小动作写入记录，下一次恢复不从空白开始。
- 最近一次放弃后，专注页提供 1/3/5 分钟恢复阶梯，并在完成记录中保留重启来源和是否计入专注。
- 专注中切到后台、锁屏、杀后台或心跳断档会立即失败，离开次数、离开时长、违规原因和时光币为 0 会进入记录和导出明细。
- 完成一段后提供“开始歇息 / 继续下一段 / 记录复盘 / 结束今天”，让用户不用自己猜下一步。
- 完成记录会自动回收到任务进度、今日目标、连续记录、项目归属和记录页。
- 记录页已按阶段分层：无记录只显示第一段引导；30 天内活跃不足 3 天只显示今日/本周和解锁提示；活跃满 3 天解锁月报；带项目归属的完成记录满 3 次才展示客户账单。

## 商业化判断

可商业化的不是计时器，而是以下结果：

- 用户减少开始阻力，形成连续专注。
- 用户能证明时间花在哪里。
- 自由职业者能把专注记录转成客户可接受的交付和对账材料。
- 长期用户能获得节奏建议和趋势复盘。

因此商业化顺序应该是：

1. 先保证每日第一段专注的完成率。
2. 再保证 7 天内有稳定记录。
3. 再提供月报、导出、账单等付费价值。

## 北极星指标

首要指标：

- 每日完成至少 1 段专注的活跃用户占比。

辅助指标：

- 新用户首次开始专注耗时。
- 首日完成专注率。
- 7 天内完成 3 天专注的比例。
- 有任务归属的专注占比。
- 月报打开率。
- 导出或账单预览触发率。

不要把“创建任务数量”“查看报表次数”“Pro 入口点击”当成主指标。它们可能上涨，但不一定说明用户核心诉求被满足。

## 下一轮产品改造重点

记录页分层已经落地（无记录引导 / 少量记录看今日本周 / 稳定记录看月报 / 项目记录足够才看账单），下一轮重点转向：

1. 首页建议继续深化：补推荐理由和本段完成后的收益提示。
2. 导出落地：把月报、账单和记录明细接入系统文件保存/分享，成为真正可交付文件。
3. 分层阈值验证：真机观察活跃 3 天解锁月报、3 次项目记录解锁账单是否符合真实节奏，必要时调整。
4. 商业能力继续后置：账单和导出只在有足够项目记录时突出，不反向渗透到首页。
